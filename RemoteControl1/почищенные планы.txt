# 1. ОБЩАЯ ИДЕЯ, КОТОРУЮ МЫ БЕРЁМ
При создании проекта менеджер выбирает:
* название
* описание
* менеджера
* участников
* **тип проекта**
* **этапы проекта**
Дальше:
* у каждого проекта свой список этапов
* у каждой задачи есть выбранный этап
* прогресс **проекта** считаем по всем задачам проекта
* прогресс **этапа** считаем только по задачам этого этапа
То есть:
**проект = большая коробка**
**этап = раздел внутри проекта**
**задача = рабочая единица внутри этапа**
# 2. ЧТО БУДЕТ ОБЩЕЕ ДЛЯ ВСЕХ ТИПОВ ПРОЕКТОВ
Независимо от модели проекта, у нас будут общие правила:
## 2.1. Проект всегда имеет этапы
Например:
* Анализ
* Backend
* Frontend
* Тестирование
или:
* Подготовка
* Разработка
* Сдача
## 2.2. Каждая задача относится к одному этапу
* задача “Сделать авторизацию” → этап Backend
* задача “Нарисовать таблицу сотрудников” → этап Frontend
## 2.3. Прогресс проекта не меняем
Он остаётся таким, как сейчас по логике:
* завершённые задачи проекта / все задачи проекта
## 2.4. У каждого этапа тоже есть свой локальный прогресс
* Backend: 3 из 5 задач завершены = 60%
* Frontend: 1 из 4 задач завершена = 25%
# 3. ВИД ПРОЕКТА №1 — ЛИНЕЙНЫЙ
Это модель, где этапы идут **по порядку**.
## Пример
* Проектирование
* Запуск
## Как это понимается
Пока не закончили прошлый этап, следующий формально ещё не основной.
## Как это выглядит в проекте
Проект: CRM для компании
Этапы:
1. Анализ
2. Проектирование
3. Разработка
4. Тестирование
5. Запуск
Задачи:
* Собрать требования → Анализ
* Описать БД → Проектирование
* Сделать backend API → Разработка
* Проверить вход и роли → Тестирование
* Подготовить релиз → Запуск
## Что удобно показывать
Для линейного проекта можно автоматически показать:
* **текущий этап** = первый этап, где ещё не всё завершено
* **следующий этап** = этап после текущего
* **завершённые этапы** = этапы, где все задачи done
## Когда подходит
Когда проект реально идёт по шагам:
* сначала понять
* потом спроектировать
* потом сделать
* потом протестировать
* потом запускать
## Плюсы
* очень понятная логика
* хорошо подходит под учебные/внедренческие проекты
* легко показывать “где мы сейчас”
## Минусы
* если работа параллельная, модель уже менее удобна
* backend и frontend в жизни часто идут одновременно
# 4. ВИД ПРОЕКТА №2 — ФУНКЦИОНАЛЬНЫЙ
Это модель, где этапы — это **не шаги по времени, а направления работы**.
* UI/UX
* QA
* Docs
Все направления могут идти параллельно.
Проект: система Remote Control
* Реализовать роли → Backend
* Переделать таблицу задач → Frontend
* Проверить отчёты → QA
* Описать структуру базы → Docs
* прогресс по каждому направлению
* сколько задач в каждом блоке
* какая часть проекта провисает
* Backend 80%
* Frontend 45%
* QA 10%
* Docs 0%
Когда команда работает параллельно по разным зонам.
* очень жизненно для реальных IT-проектов
* легко распределять задачи по ролям
* хорошо видно, что именно отстаёт
* нет понятия “текущий этап” как в линейной схеме
* сложнее показать “что идёт следующим”
# 5. ВИД ПРОЕКТА №3 — ГИБРИДНЫЙ
Это самый интересный вариант.
У проекта есть **крупные этапы сверху**, а внутри них уже **подразделы по типу работы**.
Крупные этапы:
Внутри этапа “Разработка”:
Внутри этапа “Сдача”:
* Документация
* Релиз
## Как это выглядит
Проект: платформа удалённой работы
1. Подготовка
* собрать требования
* сделать план
2. Разработка
* backend
* frontend
* ux
3. Сдача
* qa
* docs
* запуск
Задача может относиться:
* либо сразу к большому этапу
* либо к подэтапу внутри этапа
Когда проект уже посерьёзнее и хочется красивую структуру.
* очень гибко
* можно показать и порядок, и направления
* выглядит “дорого” и мощно
* самая сложная логика
* сложнее база, форма, фильтры, UI
# 6. КАК ЭТО ЛУЧШЕ СДЕЛАТЬ У ТЕБЯ
Вот мой совет.
## Не делать сразу тяжёлую иерархию 2 уровня
Потому что это уже слишком много боли на старте.
## Лучше сделать так
При создании проекта менеджер выбирает **тип проекта**:
* линейный
* функциональный
* гибрид
Но внутри данных мы пока храним всё в одном уровне:
* просто список этапов проекта
* и у задачи выбран один этап
То есть даже для гибрида на первом этапе можно сделать упрощённо.
### Например гибрид пока хранить так:
* Разработка / Backend
* Разработка / Frontend
* Разработка / UI/UX
* Сдача / QA
* Сдача / Docs
* Сдача / Релиз
То есть визуально это уже гибрид, но технически всё ещё один список этапов.
## Почему это круто
* пользователь уже получает “гибрид”
* тебе не надо делать две вложенные сущности
* потом можно развить в нормальные подэтапы

Итог такой: проект **уже не пустой и не “демка на 2 страницы”**, база у него хорошая, но до состояния **“100% проходит проверку по ТЗ без вопросов”** ещё есть заметный объём работы. По коду уже есть роли, авторизация, управление пользователями, проекты, задачи, этапы проекта, рабочий день, таймер по задаче, ручные заявки времени с вложением, отчёты и экспорты, а также заготовка календаря и настроек трекера. Самое важное: у тебя уже **есть серверная защита от “чужих задач”**. Для сотрудника сервер проверяет принадлежность задачи и при старте таймера, и при ручном учёте времени. В `OnPostStartTaskTimerAsync` и `OnPostAddManualTimeAsync` есть проверка `IsMyTaskAsync`, а в сервисе `StartTaskTimerAsync` есть ещё одна жёсткая защита: если `task.UserId != userId`, таймер не запустится. Что у тебя уже реально сделано по факту.
Авторизация и роли уже не игрушечные. Модель пользователей уже расширена под нормальный продукт: есть ФИО, логин, роль, ставка, активность аккаунта, состояние рабочего дня, настройки уведомлений, скриншотов, вебки и idle timeout. Это хороший фундамент под дальнейшую доработку, а не просто одна таблица “логин/пароль”. ([GitHub][4])
Проекты тоже уже стали умнее. В модели проекта есть менеджер, участники, `ProjectType`, `StageNamesJson`, а в миграциях видно, что этапы и типы проекта уже заведены в БД. В задачах есть `StageName`, `ProjectId`, `UserId`, дедлайн, приоритет, статус и плановые часы. Во фронте создание и редактирование задач уже завязано на проект: при смене проекта обновляются доступные исполнители и этапы. ([GitHub][5])
Рабочий день и таймер по задаче уже переведены на серверную логику. Есть `StartWorkDayAsync`, `StopWorkDayAsync`, `StartTaskTimerAsync`, `PauseTaskTimerAsync`, `StopTaskTimerAsync`; при завершении рабочего дня сервис ещё и закрывает активные task timer-сессии. Это как раз тот тип логики, который нужен для защиты от “висящих” таймеров и несогласованного состояния интерфейса. ([GitHub][6])
Ручное время уже двигается в правильную сторону. Самая большая дыра сейчас — **календарь**. В БД и `AppDbContext` календарные события уже есть, и `MainPage` их подгружает, но сам `calendar.js` при сохранении события просто делает `calendarState.events.push(...)` на фронте и перерисовывает интерфейс. То есть сейчас календарь выглядит как рабочий, но по факту ещё не живёт полноценной серверной жизнью. Для сдачи по ТЗ это риск, потому что в требованиях календарь и митапы должны быть не декоративными. ([GitHub][8])
Вторая крупная зона риска — **idle/bездействие**. База под это подготовлена: в модели пользователя есть `IdleTimeoutMinutes`, а в миграциях есть `AddIsIdleToActivityLog`. Но подтверждённой полноценной логики “мышь/клавиатура → idle → фиксация на сервере → пауза трекера/уведомление” я в коде не увидел. Значит, сейчас это именно заготовка, а не закрытый функционал. ([GitHub][4])
Третья проблема — **единый источник правды для отчётов**. Сейчас часть логики есть на сервере, но в `reports.js` и `users.js` заметная часть расчётов строится на клиенте: там собираются данные отчёта, бонусов и зарплат прямо во фронте. Это опасно: экран, CSV, Excel, PDF и “зарплатная” таблица могут начать расходиться между собой. Для тестов и защиты лучше, чтобы план/факт/бонус/зарплата считались одним серверным способом, а фронт только показывал результат. ([GitHub][9])
Четвёртая проблема — **логика прогресса проектов**. Во фронте `refreshProjectsStats()` считает общий прогресс через среднее прогрессов этапов, а пустой этап получает 0%. Это означает, что если добавить новый пустой этап или этап с новыми задачами, общий прогресс может заметно просесть — именно той проблемы ты и боишься. Плюс в `showProjectDetails()` есть опасный костыль: если у проекта нет этапов, фронт сам придумывает их по названию проекта. Для красивой демо-версии это ещё терпимо, но для надёжного продукта и тестов — плохая идея. ([GitHub][10])
Пятая зона риска — **структурное расхождение с ТЗ по данным**. В ТЗ фигурируют отдельные `WorkSessions` и `TimeEntries`, а у тебя фактически рабочая логика ушла в `ActivityLog` и `ManualTimeRequest`, плюс часть учёта идёт через task timer. Функционально это можно защитить на комиссии, но для “идеального совпадения с ТЗ” лучше либо документировать это как эквивалент, либо чуть аккуратнее привести названия и смысл сущностей к формулировкам задания. ([GitHub][8])
По твоему вопросу про ручное добавление времени: да, **в текущем виде это обязательно надо ужесточить**. Самый простой и логичный вариант для проекта — такой.
Не делать сразу полноценный бухгалтерский монстр. Делать так:
ручное время не попадает сразу в зарплату и общие часы;
сотрудник создаёт **заявку на ручное время**;
указывает задачу, часы, комментарий;
прикладывает файл, скрин, мини-отчёт или архив;
заявка получает статус `pending`;
менеджер открывает список заявок, смотрит вложение и либо `approve`, либо `reject`;
только после `approve` время учитывается в отчётах, бонусах и зарплате.
Это проще всего внедрить, потому что у тебя под это уже есть фундамент: `ManualTimeRequest`, attachment и approve/reject-обработчики уже существуют. ([GitHub][2])
Я бы довёл эту логику до такого состояния:
обязательный комментарий всегда;
вложение обязательно, если часов больше порога, например 1.5 или 2;
у заявки поля: `status`, `managerComment`, `approvedAt`, `approvedBy`;
в отчётах и зарплате учитывать только `approved`;
отклонённая заявка не удаляется, а хранится в истории;
сотрудник может открыть её, поправить и отправить заново.
Это и достаточно строго, и при этом не слишком тяжело для учебного проекта.
Теперь про три типа проектов. Ниже — как я бы это закрепил уже окончательно, простым языком.
**1. Линейный проект.**
Это когда этапы идут по порядку: сначала анализ, потом проектирование, потом разработка, потом тестирование, потом запуск. Такая модель подходит там, где работа реально идёт шаг за шагом. Пользователю всё понятно: есть текущий этап, следующий этап и завершённые этапы. Реализация у тебя простая: у проекта есть список этапов в порядке, у задачи есть `StageName`, а текущий этап — это первый этап, где ещё не всё done. Это идеально для дипломов, сайтов, внедрений, CRM и учебных проектов.
**2. Функциональный проект.**
Это когда этапы — это не шаги по времени, а направления работы: backend, frontend, design, QA, docs. Они могут идти параллельно. Здесь не нужен “следующий этап”, зато очень полезно видеть, какой блок проседает. Реализация у тебя та же самая: список этапов у проекта и `StageName` у задачи. Просто смысл этапов другой. Для Remote Control это вообще очень хороший вариант, потому что проект сам по себе уже разбивается на модули.
**3. Гибридный проект.**
Это комбинация двух подходов. Например: сверху есть крупные этапы `Подготовка`, `Разработка`, `Сдача`, а внутри можно сделать условные этапы `Разработка / Backend`, `Разработка / Frontend`, `Сдача / QA`, `Сдача / Docs`. Для пользователя это выглядит богато и “как в настоящем продукте”, а для реализации это можно пока хранить без сложной вложенности — просто как список строк. Это самый правильный вариант для твоего проекта в долгую, потому что он даёт и порядок, и параллельные направления.
Из этих трёх я бы для тебя сделал так:

внутри БД пока хранить **один список этапов**;
не делать сейчас сложные подэтапы второй сущностью;
в UI показывать:
тип проекта,
этапы,
прогресс по этапам,
текущий этап,
следующий этап только для линейного проекта.
Это даст сильный UX без перегруза архитектуры.

`decimal` тут лучше не потому, что он “моднее”, а потому что у тебя проект считает **часы, деньги, ставку, переработку, недоработку**. Для таких вещей `decimal` обычно безопаснее и стабильнее, а `double` чаще нужен для приблизительных вычислений. В твоем коде сейчас уже есть смешение типов: `HourlyRate`, `RequiredDailyHours`, `TrackedHours`, `IdleHours`, `PlannedHours`, `OvertimeHours`, `UnderworkHours` — это `decimal`, а `DurationHours` и часть VM/DTO — `double`. Из-за этого ты уже вынуждена постоянно делать приведения туда-сюда, и именно это потом рождает мелкие баги, странные округления и лишний мусор в логике.
Что по сути:
* `double` быстрее и нормален для приблизительных вычислений, графики, геометрии, процентов, где погрешность допустима.
* `decimal` лучше для денег и “человеческих” часов вида `1.25`, `7.5`, `8.00`, где нужна предсказуемость.
* У тебя предметная область именно такая, где лучше **унифицировать учет времени и оплаты на `decimal`**.

## Как я вижу правильную архитектуру проекта
У тебя проект надо разделить не “по страницам”, а **по смысловым модулям**. Тогда логика будет цельной и не поплывёт.
### 1. Модуль пользователей и ролей
Этот модуль отвечает за:
авторизацию,
регистрацию,
роли,
активность пользователя,
режим работы,
настройки профиля,
ставку,
права доступа.
То есть `User` — это не просто “кто вошёл”.
Это основная сущность сотрудника, в которой должны лежать:
ФИО, логин, email, пароль, роль, должность, ставка, активность аккаунта, режим работы, норма часов, плановое начало/конец дня, настройки трекера, idle timeout, заметки и контактные данные.

**User = кто человек, как он работает, сколько должен работать, сколько стоит его час и какие у него права.**

### 4. Модуль рабочего дня и времени
Вот это сейчас самый важный кусок.
У тебя уже фактически вырисовываются **две разные сущности времени**, и это правильно.
#### Первая сущность — рабочий день
Это не время по задаче.
Это время присутствия сотрудника в работе.

система запомнила начало,
человек нажал “Завершить рабочий день”,
система запомнила конец,
посчитала общую длительность дня.
Это нужно хранить отдельно.
#### Вторая сущность — фактически учтённое время
Это уже:
таймер по задаче,
либо одобренная ручная запись времени.
Это время, которое реально относится к задачам и может идти:
в аналитику,
в отчёты,
в зарплату,
в бонусы.
И вот это разделение у тебя **обязательно должно остаться**.
Потому что если смешать их в одну кучу, получится логическая каша:
человек просто открыл рабочий день — и уже как будто отработал задачи.
Так делать нельзя.

`manual_time` — хранит только подтверждённые ручные часы

## Как я предлагаю построить логику рабочего дня
Вот это сейчас самое главное.
Я бы закрепил **три разных понятия**.
### 1. Рабочий день
Это весь интервал:
с начала дня до конца дня.
09:00–18:00 = 9 часов рабочего дня.
Это не значит, что все 9 часов человек работал по задачам.
### 2. Учтённое время
Это время, которое реально подтверждено работой.
Туда входит:
таймер задачи,
одобренное ручное время.
по задачам набежало 5.5 часа.
### 3. Простой
Это разница между рабочим днём и учтённым временем.
рабочий день 9 часов,
по задачам учтено 5.5,
значит простой 3.5 часа.
И вот это уже логично.
Потому что у тебя получается:
**Рабочий день = присутствие**
**Учтённое время = реальная работа по задачам**
**Простой = время, когда день шёл, но задача не была запущена и работа не зафиксирована**
Это именно та модель, которую ты сейчас хочешь — и она правильная.
## Стоит ли показывать на экране:
### во сколько начался день,
### сколько уже идёт,
### сколько осталось
Да, **стоит**. Это полезно и логично.
Но показывать нужно не всё подряд всем одинаково.
### Для сотрудника
На главной в блоке рабочего дня я бы показывал:
начало дня,
текущую длительность дня,
учтённые часы по задачам,
простой,
оставшееся время до нормы — если режим фиксированный.
То есть красиво это может выглядеть так:
Начат: 09:04
В работе: 06:12
По задачам учтено: 04:50
Простой: 01:22
До нормы осталось: 01:48
Это очень наглядно.
### Для гибкого режима
Там не надо показывать “до конца рабочего дня”.
Там лучше показывать:
Сегодня отработано: 5.3 ч
До нормы осталось: 2.7 ч
или вообще:
Норма на день: свободный график
Рекомендуемый минимум: 6 ч
## Как лучше реализовать идею фиксированных и свободных сотрудников
Вот это как раз можно сделать **безболезненно**, и это реально сильная фишка.
Я бы не делал сложную систему “совсем разные типы сотрудников”.
Я бы сделал **один User**, но у него был бы `WorkMode`.
### Вариант 1. Fixed
Фиксированный режим.
У сотрудника есть:
норма часов в день,
плановое начало,
плановый конец.
Такой пользователь:
должен появиться в работу,
желательно соблюдать время,
по нему можно считать опоздания, недоработку, переработку, остаток дня.
### Вариант 2. Flexible
Гибкий режим.
У сотрудника нет жёсткого начала/конца дня.
Есть только:
норма часов в день или в неделю,
дедлайны задач,
факт выполнения.
сам решает, когда начать,
сам решает, когда закончить,
главное — выполнить норму и задачи.
Это вообще не ломает проект.
Это просто другое поведение интерфейса и отчётов.
## Как это реализовать проще всего
Вот самый лёгкий и правильный путь.
### В `User` оставить:
`WorkMode`
`RequiredDailyHours`
`PlannedStartTime`
`PlannedEndTime`
### Логика:
если `WorkMode = fixed`,
то:
показываем начало/конец,
показываем остаток до конца дня,
можно считать опоздание и переработку.
если `WorkMode = flexible`,
не требуем начала ровно по часу,
не показываем “смена до 18:00”,
показываем только:
сколько уже отработано,
сколько осталось до нормы,
какие дедлайны горят.
Вот и всё.
То есть архитектурно это не отдельный тип таблицы и не другой модуль.
Это **просто другой режим поведения одного и того же пользователя**.
И это очень хорошо, потому что не ломает ничего.
## Есть ли смысл делать назначение начала/конца дня через менеджера
Да, но только как настройку, а не как ежедневное ручное назначение.
То есть менеджер один раз задаёт сотруднику режим:
либо фиксированный график 09:00–18:00 и 8 часов,
либо гибкий режим и, например, 6 или 8 обязательных часов,
либо вообще просто дедлайны без строгого окна.
Это лучше, чем каждый день руками назначать смену.
Почему:
иначе менеджер будет закапываться в рутине;
логика станет тяжёлой;
интерфейс станет перегруженным.
Поэтому лучший вариант:
**менеджер задаёт шаблон режима работы сотрудника**
а не вручную рулит каждым днём.
Потом уже, если захочешь, можно сделать исключения:
сегодня смена с 10:00,
завтра выходной,
в пятницу короткий день.
Но это уже потом.
## Как должна выглядеть правильная серверная логика по рабочему дню
И вот тут главное: да, **основная логика должна быть на C#**, не на JS.
Я бы разделил так.
### На C# должны жить:
старт рабочего дня,
завершение рабочего дня,
старт/пауза/стоп таймера,
проверка принадлежности задачи пользователю,
расчёт рабочего дня,
расчёт учтённых часов,
расчёт простоя,
расчёт переработки/недоработки,
расчёт остатка до нормы,
зарплата,
бонусы,
отчёты,
права доступа,
одобрение ручного времени,
логика проектов и этапов.
### На JS должны жить:
отрисовка,
обновление экрана,
открытие модалок,
фильтры,
анимации,
переключение вкладок,
вызовы API.
То есть фронт — это оболочка, а не источник правды.
Это для тебя сейчас самый правильный курс.

# ЭТАП 1. Стабилизировать ядро времени
Сначала добиваем логику рабочего дня и времени.
Что именно делаем:
закрепляем разделение:
рабочий день отдельно,
фактическое время отдельно;
проверяем, что:
рабочий день стартует только один раз,
второй раз стартовать нельзя,
при завершении дня таймеры закрываются,
таймер нельзя стартовать без рабочего дня,
чужую задачу нельзя выбрать ни фронтом, ни сервером;
доделываем расчёты:

`trackedHours`,
`idleHours`,

`underworkHours`;

Это критически важно.

# 2. Что у тебя должно быть как минимально завершённое ядро

## Модуль A. Пользователи, роли, профиль

### Что добавить для законченности

## Модуль B. Рабочий день и учёт времени
Вот это твой самый главный модуль.
Нужно окончательно закрепить 3 сущности:
### Рабочий день
Это весь период присутствия в работе.
### Учтённое время
Это только то, что реально относится к задачам:
таймеры + одобренное ручное время.
рабочий день − учтённое время.
Это уже очень логичная модель. И да, на главной **обязательно стоит показывать**:
во сколько начался день,
а для fixed ещё и сколько до конца смены.
Это реально полезно, а не “лишняя красота”.
### Что добить здесь
убрать остатки старой логики,
считать всё через C#,
унифицировать `decimal`,
разделить:
`salaryHours`.
## Модуль C. Ручной учёт времени
Вот тут у тебя уже есть очень сильная база, и её надо не испортить, а довести до красивой схемы.
## Как это должно работать
Сотрудник не “добавляет себе часы напрямую”.
Он создаёт **заявку**.
### Заявка содержит:
задачу,
часы,
комментарий,
вложение при необходимости,
причину ручного добавления.
### Дальше у заявки есть статусы:
`draft`
`submitted`
`approved`
`rejected`
`needs_revision`
И вот это место с `needs_revision` — как раз твоя классная идея.
## Логика возврата на доработку
Менеджер может не только отклонить, но и:
вернуть заявку на доработку,
написать комментарий,
попросить:
добавить файл,
исправить часы,
уточнить, что именно сделано,
добавить доказательство.
То есть не сразу “отказ”, а **нормальный цикл согласования**.
### Как это выглядит для пользователя
Сотрудник отправил заявку.
Менеджер пишет:
“Добавь скрин результата и уточни, почему 4 часа”.
Заявка получает статус `needs_revision`.
Сотрудник редактирует её и отправляет повторно.
Только после `approved` создаётся `ActivityLog(manual_time)`.
Вот это очень хорошая и уже почти продуктовая логика.
### Что ещё сюда добавить
обязательный комментарий всегда,
вложение обязательно после порога, например от 2 часов,
история изменений заявки,
кто и когда проверил,
комментарий менеджера,
повторная отправка,
фильтр “ожидает проверки / на доработке / одобрено / отклонено”.
## Модуль D. Проекты и этапы
Вот тут у тебя уже есть сильная идея, и её обязательно надо довести.
## Типы проектов
### Линейный
этапы идут по порядку.
### Функциональный
этапы — это направления: backend, frontend, qa и т.д.
### Гибридный
крупные этапы + функциональные подблоки, но пока храним это как один плоский список этапов.
Это очень хорошая модель и она реально похожа на то, как разные системы организуют работу: Jira делает упор на workflow и переходы, Asana — на проекты и портфели, Notion — на гибкие базы и виды, monday — на обзор и загрузку. ([Atlassian Support][1])
## Что обязательно должно быть
у проекта есть тип,
у проекта есть этапы,
у задачи есть этап,
у этапа есть локальный прогресс,
у проекта есть общий прогресс,
для линейного проекта есть:
следующий этап,

## Что добавить для силы
фильтр по этапу,

блоки этапов внутри проекта как в Asana,
прогресс этапа отдельной полоской,
этап с риском,
этап без задач,
этап с просрочками.
### Очень важная правка логики
Надо убрать ситуацию, когда новый пустой этап слишком роняет общий прогресс проекта.
Лучше считать проект:
либо по задачам,
либо по этапам, но так, чтобы пустой этап не ломал всю картину.

Вот это обязательно надо перевести на **один серверный источник правды**.
## Что считать на C#
плановые часы,
рабочие часы,
простои,
переработки,
недоработки,
зарплатные часы,
сумму к оплате,
рекомендацию по бонусу.
### Какие отчёты должны быть
по сотруднику,
по проекту,
по периоду,
по этапам проекта,
по простоям,
по ручным заявкам,
по просрочкам,
по зарплате.
Asana и monday прямо усиливают продукты через дашборды и workload-обзор по нескольким проектам и сотрудникам. У monday dashboards и workload-виджеты на это и заточены; у Asana portfolios и workload тоже решают именно эту задачу. ([Asana][3])
### Какую бонусную логику сделать простой и понятной
Сейчас не надо усложнять.
Делать так:
если задача завершена без просрочки и факт не вышел за предел разумного отклонения — хороший результат,
если сотрудник стабильно закрывает план и мало idle — “рекомендуется бонус”,
если просрочки и большой idle — без бонуса.
Пока это можно выводить именно как текстовую рекомендацию, без сложной бухгалтерии.
