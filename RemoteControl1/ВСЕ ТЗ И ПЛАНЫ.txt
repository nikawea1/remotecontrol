## Риск 2. В `MainPage.cshtml.cs` русские строки выглядят как CP1251/битая кодировка

В терминале часть русских строк в `MainPage.cshtml.cs` отображается как `\xcf...`, `\xd4...` и т.п. Это может быть просто особенностью кодировки файла, но может привести к кривым русским сообщениям/заголовкам в экспортах или ошибках.

**Не ломать сейчас.**
Но перед финалом надо проверить:

* что ошибки в браузере читаемые;
* что CSV/Excel/Word/PDF открываются с нормальными русскими заголовками;
* что файл сохранён в нормальной UTF-8 кодировке.

---

## Риск 3. Остался старый прямой endpoint ручного времени

В UI быстрый учёт уже отправляет заявку через `CreateManualTimeRequest`, это хорошо. Но в `MainPage.cshtml.cs` ещё есть `OnPostAddManualTimeAsync`, а в `TaskService` есть `AddManualTimeAsync`, который создаёт `manual_time` напрямую.

**Почему это риск:**
Даже если фронт его не использует, сам endpoint остаётся потенциальным обходом согласования.

**Правильное простое решение потом:**

* либо полностью убрать прямой endpoint;
* либо оставить только для admin/manager;
* либо сделать так, чтобы он тоже создавал заявку, а не `ActivityLog(manual_time)` напрямую.

**Статус:** функционально не мешает, но архитектурно это дыра против твоего правила “ручное время только после approve”.

---

## Риск 4. Проект удаляется слишком жёстко

В ТЗ допускается удаление проекта, если к нему не привязаны задачи, или архивирование. По текущей логике удаление проекта может снести связанные задачи, activity logs, заявки ручного времени и участников.

**Почему это опасно:**
Для учебного проекта это может сработать, но для нормальной логики это слишком разрушительно.

**Лучше потом:**

* заменить удаление на архивирование;
* физическое удаление разрешать только если нет задач/логов/заявок;
* сделать `IsActive=false` или `IsArchived=true`.

## Риск 6. Users page доступна manager, но кнопка “Добавить сотрудника” видна без Razor-ограничения

`Users.cshtml.cs` пускает на страницу admin и manager. Но `OnPostAddUserAsync`, `OnPostUpdateUserAsync`, `OnPostToggleUserStatusAsync` разрешают действия только admin. Это правильно на сервере.

**Простой фикс потом:**
Для manager скрыть или disabled-кнопку с подсказкой.

* запрет редактирования роли/ставки сотрудником;
* сохранение телефона/должности/заметок;
* смена пароля: старый неверный, новый короткий, подтверждение;
* настройки трекера реально сохраняются в БД;
* после обновления страницы значения не сбрасываются.

**Статус:** модуль уже живой, но нужен полный ручной прогон.

# 6. Подробно по вкладкам проекта

## 6.1. Главная / Dashboard

### Элементы, которые есть

**1. Рабочий день**

* старт рабочего дня;
* завершение рабочего дня;
* проверка активного рабочего дня с сервера;
* рабочий день хранится как `ActivityLog(workday)`;
* при завершении дня активные таймеры закрываются;
* считается длительность дня;
* считается tracked/idle/overtime/underwork.

**2. Таймер задачи**

* выбор задачи;
* старт;
* пауза;
* стоп;
* запрет старта без рабочего дня;
* запрет чужой задачи;
* активный timer хранится как `ActivityLog(task_timer)`.

**3. Быстрый учёт времени**

* задача;
* дата;
* часы;
* причина;
* комментарий;
* файл;
* заявка уходит в `ManualTimeRequest`;
* повторная отправка после `needs_revision` уже поддерживается.

**4. Мои задачи / ближайшие задачи**

* вывод текущих задач пользователя;
* статусы;
* дедлайны;
* переход к задачам.

**5. Последняя активность**

* активность берётся из logs;
* связана с таймерами и ручным временем.

**6. KPI-карточки**

* часы;
* задачи;
* активность;
* базовая аналитика.

### Что не добито

* динамические мини-тренды на карточках: `+8 ч`, `-2 задачи`, стрелки вверх/вниз;
* сравнение с прошлым периодом;
* красивый блок “сегодня горит”;
* блок “скоро дедлайн”;
* блок “последние уведомления”;
* более явное разделение: рабочий день / учтённое время / простой;
* расширенная заявка ручного времени лучше должна жить ещё и во вкладке “Задачи”.

### Рекомендация

На главной оставить **короткий виджет быстрого времени**, а во вкладке “Задачи” сделать **полноценную раскрывающуюся панель ручной заявки**. Так будет логично:

* главная — быстро отправить;
* задачи — подробно работать с конкретной задачей, файлами, комментариями и статусом заявки.

**2. Фильтры**

**3. Создание задачи**

  * задача из этапа проекта создаётся именно в выбранном этапе;
  * редактирование этапа не ломает project view;
  * employee может менять только свой статус.

### Статус

Задачи — один из самых сильных модулей сейчас. Основа хорошая. Следующий уровень — комментарии, файлы, история и более красивый просмотр.

## 6.3. Проекты

ТЗ требует список проектов, поиск, сортировку, создание проекта, редактирование и статистику. 

### Что есть

**1. Карточки проектов**

* название;
* описание;
* менеджер;
* количество задач;
* текущий этап;
* прогресс;
* этапы;
* участники компактно;
* кнопки действий скрываются для employee.

**2. Создание проекта**

* название;
* описание;
* менеджер;
* тип проекта;
* участники;
* этапы;
* кастомный выбор участников через picker;
* этапы можно добавлять, удалять, переименовывать;
* этапы можно двигать вверх/вниз;
* есть перенос этапа в начало/конец.

**3. Типы проекта**

* linear;
* functional;
* hybrid.

**4. Этапы**

* `StageNamesJson`;
* у задач есть `StageName`;
* просмотр проекта группирует задачи по этапам;
* прогресс этапа считается отдельно.

**5. Прогресс**

* общий прогресс проекта считается по задачам;
* этапы имеют собственный прогресс;
* пустой этап не должен ломать общий процент проекта.

**6. Добавить задачу в этап**

* кнопка есть;
* функция есть;
* проект и этап передаются в модалку задачи.

### Что не добито

* архивирование проекта вместо разрушительного удаления;
* риски проекта;
* “здоровье проекта” зелёный/жёлтый/красный;
* просрочки по этапам;
* текущий/следующий этап надо лучше адаптировать под тип проекта:

  * linear — текущий/следующий этап логичны;
  * functional — лучше “самый активный / самый проблемный этап”;
  * hybrid — можно показывать “активный блок”.
* drag & drop задач между этапами;
* история проекта;
* комментарии и файлы проекта;
* шаблоны проектов;
* копирование проекта.

### Важная рекомендация по “готовности”

Не возвращать второй процент “Готовность”, если он дублирует общий прогресс.

Лучше так:

* **Общий прогресс проекта** — done задачи / все задачи.
* **Текущий этап** — название + прогресс этапа.
* Для linear: ещё “Следующий этап”.
* Для functional/hybrid: “Проблемный этап” или “Активный этап”.

Так будет логичнее, чем два одинаковых процента.

### Статус

Проекты уже сильно доработаны. Сейчас это не слабый модуль. Но его надо стабилизировать ручной проверкой и потом довести UX.

## 6.4. Трекер

ТЗ требует отдельную страницу трекера с крупным таймером, выбором задачи, кнопками старт/пауза/стоп и историей рабочих сессий. 

### Что есть

* страница `Tracker`;
* крупный таймер;
* выбор задачи;
* старт/пауза/стоп;
* рабочий день;
* история рабочих дней;
* синхронизация с серверным статусом рабочего дня;
* idle timeout на фронте;
* авто-пауза при простое;
* bottom tracker в layout.

### Что важно

Фронтовый idle уже есть как поведение: если нет активности, таймер может уйти на паузу. Но полноценного серверного журнала idle-интервалов пока нет.

ТЗ отдельно говорит про трекинг активности мыши/клавиатуры и простой при отсутствии ввода. 

### Что не добито

* отдельные `ActivityLog(idle)` интервалы;
* серверный endpoint фиксации idle;
* отчёт по простоям;
* уведомление менеджеру о долгом простое;
* всплывашка “Вы всё ещё работаете?”;
* нормальная политика: auto-pause или просто фиксировать idle;
* просмотр скриншотов менеджером.

### Статус

Трекер работает как базовый модуль. Idle пока “полуготовый”: поведение есть, полноценного продукта контроля простоя ещё нет.

## 6.5. Быстрый / ручной учёт времени

ТЗ требует ручное добавление времени по задачам: задача, часы, комментарий. 
В твоих планах ручное время должно быть строже: заявка, файл, комментарий менеджера, возврат на доработку и только approved в расчёты. 

### Что есть

* `ManualTimeRequest`;
* заявка;
* статус `pending`;
* `approved`;
* `rejected`;
* `needs_revision`;
* комментарий сотрудника обязателен;
* файл обязателен;
* manager/admin может approve/reject/return for revision;
* после approve создаётся `ActivityLog(manual_time)`;
* только после approve время становится фактом.

### Что уже хорошо

Это сильнее, чем минимальное ТЗ. Ты правильно ушла от схемы “сотрудник сам себе начислил часы”.

### Что не добито

* UX возврата на доработку;
* история изменений заявки;
* редактирование заявки после возврата надо проверить руками;
* комментарий менеджера должен красиво отображаться сотруднику;
* фильтры по статусу заявок;
* порог вложения: сейчас файл фактически обязателен всегда, а ты хотела гибче — например, обязательно от 2 часов;
* отдельный список “мои заявки” во вкладке задач;
* отчёт по ручным заявкам.

### Главная рекомендация

Не переносить быстрый учёт времени полностью с главной.

Лучше:

* на главной — короткий виджет;
* во вкладке “Задачи” — расширенная панель “Ручная заявка времени”;
* в карточке задачи — кнопка “Запросить ручное время”;
* в разделе сотрудников/админке — таблица заявок на проверку.

* выгрузка строго по текущему выбору фильтров;

ТЗ ГЛАВНЫЕ ТРЕБОВАНИЯ:
41. Платформа для управления удаленными сотрудниками
Функционал:
● Учет рабочих часов через трекер активности (скриншоты, время за
задачами).

● Распределение задач по проектам с приоритетами и дедлайнами.
● Интеграция с календарями для планирования встреч и митапов.
● Аналитика продуктивности: сравнение плановых и фактических затрат
времени.
● Система бонусов за перевыполнение целей.
● Уведомления о переработках или простоях.
● Генерация отчетов для расчета зарплаты.



ТЕХНИЧЕСКОЕ ЗАДАНИЕ:
1. Общие сведения
Программное средство «REMOTE CONTROL» – это веб‑приложение для
управления удалёнными сотрудниками, учёта их рабочего времени и базового
управления задачами и проектами. Приложение работает в браузере, серверная
логика реализуется на языке программирования C#, а пользовательский
интерфейс создаётся с использованием HTML, CSS и JavaScript.
1.1 Формулировка задания
Разработать и реализовать веб-приложение «REMOTE CONTROL» и основные
функциональные модули, которое позволяет руководителю управлять задачами
удалённых сотрудников и учитывать их рабочее время, а сотрудникам удобно
отмечать отработанные часы и видеть свои задачи и встречи. Реализовать
модули и функции обеспечивающие: учёт рабочих часов удалённых
сотрудников через трекер активности (запуск/остановка рабочего дня и таймер
времени по задаче); распределение задач по проектам с указанием приоритетов,
дедлайнов и планового времени; планирование встреч и митапов через
встроенный календарь; аналитику продуктивности на уровне плановых и
фактических затрат времени по задачам и проектам; простую систему бонусов
за перевыполнение целей (текстовые пометки в отчётах); уведомления о
переработках и недораб; генерацию отчётов по времени и оплате для расчёта
заработной платы. Дополнительно реализовать базовый функционал
регистрации, авторизации и разграничения прав пользователей
(сотрудник/руководитель), а также простой личный кабинет с просмотром
профиля.
1.2 Цели, достигаемые разработкой
Создание базового инструмента для контроля фактического рабочего времени
удалённых сотрудников через веб‑интерфейс. Обеспечение управления
задачами: создание задач, назначение исполнителей, приоритеты и сроки
исполнения. Реализация формирования отчётов по затраченному времени и
расчёта сумм к оплате на основе часовой ставки. Предоставление наглядной
аналитики по отклонениям между плановым и фактическим временем по
задачам, сотрудникам и проектам. Обеспечение минимальной системы бонусов
за перевыполнение плановых показателей и простых уведомлений о
переработках или простоях.
1.3 Категории пользователей
Программное средство «REMOTE CONTROL» должно быть рассчитано на два
типа пользователей:
Сотрудник:
 регистрируется в системе и входит по логину и паролю;
 запускает и останавливает рабочий день через веб‑интерфейс (кнопки
«Начать рабочий день» / «Завершить рабочий день»);

 запускает и останавливает трекер времени по выбранной задаче (кнопки
«Старт», «Пауза», «Стоп» в блоке трекера);
 вручную добавляет записи о затраченном времени по задачам (выбор
задачи, часы, комментарий);
 просматривает список назначенных задач, фильтрует их по статусу и
приоритету, ищет по названию, детали задачи;
 может изменять статус своих задач;
 просматривает историю своих временных записей и последних
активностей;
 просматривает простые показатели по себе на главной панели
(отработано часов, количество активных и выполненных задач);
 при необходимости просматривает свой профиль (ФИО, логин) и может
сменить пароль.
Руководитель (администратор):
 входит в систему с ролью «администратор» с помощью уникального
логина и пароля;
 управляет учётными записями сотрудников: создаёт новые учётные
записи (логин, пароль, ФИО, роль, часовая ставка), редактирует и при
необходимости деактивирует пользователей;
 создаёт проекты, редактирует их и просматривает список проектов с
краткой статистикой (количество задач, процент готовности);
 создаёт задачи в модальном окне: задаёт название, описание, проект,
исполнителя, приоритет, плановое время и дедлайн;
 просматривает общий список задач, использует фильтры по статусу,
приоритету, проекту и поиск по названию;
 просматривает отчёты: суммарное отработанное время, количество
выполненных и просроченных задач, эффективность, распределение
часов по дням недели и по проектам;
 формирует отчёт по задачам: плановые и фактические часы, отклонение,
статус задачи и краткая сводка по проектам;
 на основе отчётов получает данные для расчёта зарплаты и оценки
бонусов.

2. Описание предметной области
В условиях роста удаленной занятости руководителям необходимы простые и
прозрачные инструменты для учета рабочего времени сотрудников.
Электронные таблицы (Excel) не обеспечивают автоматической фиксации
активности, контроля соответствия задач и времени, а также оперативного
формирования отчетов. Процессы, которые должны включаться в программный
продукт «REMOTE CONTROL»:

 фиксировать начальное и конечное время рабочего дня сотрудника;
 отслеживать, сколько часов реально потрачено на конкретные задачи в
рамках проектов;
 видеть список задач с приоритетами и сроками выполнения;
 сравнивать плановое и фактическое время по задачам и проектам;
 быстро формировать отчёты для расчёта оплаты труда и оценки
продуктивности.
Использование только электронных таблиц (Excel и аналогов) приводит к
ручному вводу данных, сложности в контроле соответствия задач и времени, а
также к трудностям при формировании аналитических отчётов.
Веб‑приложение «REMOTE CONTROL» решает эти проблемы за счёт
централизованного хранения данных, удобного интерфейса для сотрудников и
руководителя.
2.1 Описание вычислительного процесса
Бизнес-процессы программного средства «REMOTE CONTROL»:
Разработка приложения: Разработчик создает приложение на основе
настоящего технического задания.
Работа сотрудника: Сотрудник запускает трекер, работает за компьютером,
периодически вручную указывает, над какой задачей работал. Система
фиксирует активность, делает снимки.
Работа руководителя: Руководитель создает задачи, назначает их,
просматривает отчеты и статистику.
2.2 Состав данных и алгоритмы обработки информации
Разрабатываемый программный продукт будет создавать и модифицировать
базу данных, содержащую следующие таблицы:
Users (Пользователи): Хранит уникальный логин, пароль, ФИО, должность,
часовую ставку и роль пользователя (employee/manager).
Projects (Проекты): Содержит ID, название и описание проекта.
Tasks (Задачи): Содержит данные о задачах: название, описание, ID проекта, ID
исполнителя (сотрудника), приоритет (1-низкий, 2-средний, 3-высокий), статус,
плановое время (часы), дата дедлайна.
WorkSessions (Рабочие сессии): Фиксирует автоматические сессии: ID
сотрудника, время начала/окончания, общее время активности, список путей к
снимкам экрана.

TimeEntries (Учет времени): ID сотрудника, ID задачи, дата, затраченные часы,
комментарий.
Для обработки информации будут использоваться следующие алгоритмы:
Аутентификация: Сравнение введенного пароля с паролем, хранящимся в БД.
Трекинг активности: Фоновый таймер, отслеживающий пользовательский ввод
(мышь/клавиатура). При отсутствии ввода в течение N секунд период считается
простоем.
Создание скриншотов: Алгоритм снимка экрана с заданным интервалом в
периоды активности.
Формирование отчета: Алгоритм выборки и обработки данных из таблиц
TimeEntries, Tasks, Users за выбранный период с расчетом отклонений и сумм.
2.3 Недостатки существующих проектных решений
Существующие аналогичные системы (TimeDoctor, Toggl Track) часто
являются облачными, имеют избыточный функционал и высокую стоимость
подписки. Локальное desktop-приложение с открытым для модификации кодом,
сфокусированное на базовых нуждах малого бизнеса, удовлетворяет
потребность в недорогом, контролируемом и конфиденциальном решении.

3.3 Функциональные требования
Для Сотрудника:
1. Регистрация и авторизация
- Система должна предоставлять форму входа, где сотрудник указывает логин и
пароль.
- При успешной проверке данных пользователь перенаправляется на главную
страницу.
- В зависимости от роли (сотрудник/руководитель) доступны разные разделы
(например, сотрудник не видит управление пользователями).
- При неправильном логине или пароле отображается понятное сообщение об
ошибке.
2. Трекер рабочего дня
- На главной странице отображается блок «Рабочий день» с двумя состояниями:
- если рабочий день не начат — кнопка «Начать рабочий день»;
- если день начат — кнопка «Завершить рабочий день» и информация о
текущем времени в работе.
- При нажатии «Начать рабочий день»:

- создаётся запись рабочего дня с текущей датой и временем начала;
- интерфейс переключается в состояние «день начат».
- При нажатии «Завершить рабочий день»:
- в записи рабочего дня сохраняется время окончания;
- автоматически рассчитывается количество отработанных часов за день;
- запись сохраняется в историю рабочих дней.
- На странице трекера или в отдельном блоке должна отображаться таблица
«История рабочих дней» (дата, начало, конец, длительность).
3. Трекер времени по задаче
- В блоке «Трекер времени» сотрудник выбирает задачу из выпадающего
списка.
- Доступны кнопки «Старт», «Пауза» и «Стоп»:
- «Старт» запускает таймер отсчёта времени по выбранной задаче;
- «Пауза» временно останавливает отсчёт без сброса;
- «Стоп» останавливает таймер и фиксирует длительность сессии.
- После нажатия «Стоп» система может автоматически создать запись учёта
времени (TimeEntry) с текущей датой, задачей и количеством часов,
рассчитанных по таймеру.
- Сотрудник может видеть текущий отработанный интервал в формате
«ЧЧ:ММ:СС».
4. Учёт времени по задачам
- На главной или отдельной странице есть блок «Быстрый учёт времени»:
- выпадающий список задач, назначенных сотруднику;
- поле «Часы» (разрешены дробные значения, например 0.5, 1.5);
- поле «Комментарий» (что именно было сделано);
- кнопка «Добавить».
- При нажатии «Добавить»:
- создаётся запись в таблице TimeEntries (пользователь, задача, дата, часы,
комментарий);
- запись появляется в таблице последней активности.
- На странице может быть таблица «Мои записи времени»: дата, задача, часы,
комментарий.
5. Просмотр и управление задачами
- Сотрудник видит страницу «Задачи», где отображаются все задачи,
назначенные этому пользователю.
- Каждая задача содержит: название, проект, приоритет, статус, дедлайн,
плановые часы.
- Доступны фильтры:
- по статусу (Все, Новые, В работе, На проверке, Завершённые);

- по приоритету (Все, Высокий, Средний, Низкий).
- Доступно поле поиска по названию задачи.
- При выборе задачи сотрудник может изменить её статус (например, с «Новая»
на «В работе», затем на «На проверке» или «Завершена») и при необходимости
добавить комментарий.
6. Просмотр аналитики и личной статистики
- На главной странице сотрудник видит:
- общее количество отработанных часов за выбранный период;
- количество активных задач;
- количество выполненных задач;
- количество просроченных задач.
- Эти показатели можно выводить в виде карточек (как в твоём макете).
7. Профиль пользователя (простое доп. требование)
- Сотрудник может открыть страницу «Профиль», где отображаются: ФИО,
логин, роль, часовая ставка.
- На этой же странице сотрудник может сменить пароль, указав старый и новый
пароль.

Для руководителя:
1. Управление пользователями
- Страница «Сотрудники» (может быть доступна только руководителю):
- таблица с логином, ФИО, ролью, часовой ставкой, активностью
(активен/неактивен);
- кнопка «Добавить пользователя» для открытия формы создания;
- возможность редактировать существующие записи (ФИО, роль, ставка,
пароль);
- возможность деактивировать учётную запись (без физического удаления).
- При создании пользователя нужно указать минимум: логин, пароль, ФИО,
роль, часовую ставку.
2. Управление проектами
- Страница «Проекты»:
- список проектов в виде карточек;
- кнопка «Новый проект» (открывает форму с полями «Название» и
«Описание»);
- возможность редактировать проект (менять название и описание);

- возможность удалить проект, если к нему не привязаны задачи (или просто
помечать как архивный).
3. Управление задачами
- Руководитель может создавать задачи через форму (например, модальное
окно):
- проект;
- название;
- описание;
- исполнитель (выбор из списка пользователей с ролью «сотрудник»);
- приоритет (низкий, средний, высокий);
- плановое время (часы);
- дедлайн (дата).
- Руководитель видит общий список задач, может фильтровать их по проекту,
исполнителю, статусу и приоритету.
- Руководитель может изменять основные параметры задачи, включая статус и
исполнителя.
4. Аналитика продуктивности (план и факт)
- На странице отчётов и/или на главной панели для руководителя
отображаются:
- суммарные плановые часы по задачам;
- суммарные фактические часы (по записям TimeEntries);
- отклонения по задачам и сотрудникам (факт − план);
- количество выполненных и просроченных задач.
- Информация представляется в виде таблиц и простых визуальных элементов
(полоски‑барчарты, прогресс‑бары), без сложных диаграмм.
5. Система бонусов за перевыполнение целей
- В отчёте по задачам добавляется колонка «Бонус»:
- если фактические часы по задаче или по сотруднику за период превышают
плановые на установленный порог (например, 10%), выводится текст
«Рекомендуется бонус»;
- в остальных случаях — «Без бонуса».
- Никакого отдельного журнала бонусов или сложных формул не требуется.
6. Уведомления о переработках и простоях
- В отчёте и/или на панели руководителя отображается:
- список «Переработки» — задачи/сотрудники, у которых факт &gt; план;
- список «Недозагружены» — задачи/сотрудники, у которых факт сильно &lt;
плана.

- Уведомления могут быть реализованы в виде небольших информационных
блоков или всплывающего блока на странице, без сложных push‑уведомлений
и звуков.
7. Генерация отчётов для расчёта зарплаты
- Руководитель выбирает период (дата «с» и «по») и сотрудника (или «Все
сотрудники»).
- Система формирует таблицу: ФИО, фактические часы за период, часовая
ставка, сумма к оплате.
- Таблица может быть выгружена в CSV (отчёт для Excel) или напечатана из
браузера.
3.4 Требования к интерфейсу
Интерфейс должен быть реализован в виде адаптивного веб‑сайта (как в твоём
макете: верхнее меню, контейнер, карточки, таблицы и т.д.).
Все основные действия должны выполняться в 1–2 клика из соответствующих
разделов.
Навигация между разделами («Главная», «Задачи», «Проекты», «Трекер»,
«Отчёты») осуществляется через верхнее меню.
Для форм использовать понятные подписи, плейсхолдеры и простые сообщения
об ошибках (например, «Заполните поле Название задачи»).
3.5 Требования к алгоритмам
Простота использования: Интерфейс должен быть интуитивно понятным.
Основные действия выполняются в 1-2 клика.
Функциональность: Интерфейс предоставляет быстрый доступ к основным
функциям через главное меню и панели инструментов.
Адаптивность: Окна приложения должны корректно масштабироваться,
основные контролы (таблицы, кнопки) оставаться доступными.
3.6 Прочие требования
Алгоритм трекинга: Должен работать в фоновом режиме с минимальным
потреблением ресурсов CPU.
Алгоритм работы с БД: Операции чтения/записи не должны блокировать
интерфейс (использовать фоновые вызовы).
Алгоритм создания отчетов: Формирование стандартного отчета за месяц (для
10 сотрудников) должно занимать не более 5 секунд.
4. Порядок контроля и обеспечение качества

4.1 Экспертиза
Не требуется разработка макета на стадии «Технический проект». Требуется
проведение экспертизы готового программного продукта при участии
представителя Заказчика. Экспертиза включает проверку соответствияы

4.2 Тестирование
Тестирование должно проводиться Разработчиком и включать этапы:
Модульное тестирование: аутентификации, алгоритмов расчета времени,
формирования SQL-запросов.
Интеграционное тестирование: взаимодействия модуля трекинга, БД и UI.
Системное тестирование на соответствие всем функциональным требованиям
раздела 3.3.
Тестирование проводится на тестовой базе данных с демонстрационными
данными.
4.3 Опытная эксплуатация
Для программного продукта «REMOTE CONTROL» проведение опытной
эксплуатации во внешних организациях не требуется. Финальное приемо-
сдаточное тестирование проводится Заказчиком.
5. Требования к документированию
5.1 Требования к справочной подсистеме
В связи с ориентированностью на простоту, сложная справочная система не
требуется. Прототип должен содержать:
Краткие подсказки для основных элементов управления.
Понятные сообщения об ошибках (например, &quot;Не удалось подключиться к базе
данных&quot;, &quot;Неверный логин или пароль&quot;).
Комментарии в коде для основных алгоритмов и функций.
5.2 Требования к документации пользователя
Руководство сотрудника: Краткая инструкция по запуску трекера, ручному
учету времени и отправке отчета.
Руководство руководителя: Описание функций создания задач, назначения
исполнителей и работы с модулем отчетности.
Описание структуры базы данных: Документация для разработчиков с
описанием таблиц, полей, связей и примеров запросов.
